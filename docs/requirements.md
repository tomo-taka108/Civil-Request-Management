## 1. システム概要

### 1.1 背景・目的

地方の土木事務所では、道路・河川・砂防施設等の土木インフラに関する苦情・要望を、窓口・電話・メール・要望書など複数の手段で日々受け付けている。また、職員のパトロールによって発見される施設の異常箇所も別途対応が必要となる。

現状、これらの情報が一元管理されておらず、以下のような課題が想定される。

- 受付情報が担当者・部署ごとに分散し、全体像の把握が困難
- 対応状況（未対応／対応中／対応完了）の進捗管理が煩雑
- 過去の要望・対応履歴の検索・参照に時間がかかる
- 要望箇所の地理的な偏在や傾向の分析ができない

本システムは、これらの苦情・要望・異常箇所情報を一元管理し、受付から対応完了までの進捗を可視化することで、業務の効率化と対応品質の向上を図ることを目的とする。

### 1.2 システム化の対象範囲

- 対象施設：道路（橋梁・トンネルを含む）、河川、砂防施設
- 対象事案：住民等からの苦情・要望、および職員パトロールによる異常発見

### 1.3 想定利用者・権限方針

- 土木事務所内の全職員が、全案件を閲覧できることを基本とする（部署を横断して状況を把握できる点に意味がある）
- 編集・削除（対応状況の更新、対応方針の登録等）は、案件の対応部署に応じた担当者のみ可能とする（道路案件は道路担当、河川案件は河川担当が編集）
- 新規登録（受付担当・パトロール担当）は、担当部署によらず職員であれば誰でも可能とする
- 利用は事務所内に閉じたクローズドな環境であり、A事務所の職員がB事務所の案件を閲覧することはない前提のため、ID・パスワードによるログイン認証は必須とする（詳細は「4.1 セキュリティ・個人情報保護」参照）
- 事務所ごとのデータ分離方式：単一システム上でユーザーアカウントに「所属事務所」を紐づけ、ログイン時にシステム側が所属事務所を自動判定する方式とする（事務所ごとにURL・インフラを分離する方式は採らない）。既存RDSへの相乗りにより追加コストを抑える方針（4.3参照）とも整合する
- ユーザーアカウントは職員本人による新規登録は行わず、システム管理者権限を持つ職員が発行する。権限区分は「一般職員」「システム管理者」の2種類とし、システム管理者はユーザーアカウントの登録・編集・無効化（異動・退職時）を行える

## 2. 機能要件

### 2.1 苦情・要望・異常箇所の登録／削除／修正

- 窓口・電話・メール・要望書・FAX・職員パトロール・その他など、受付経路によらず同一の仕組みで登録できること
- 登録済みデータの修正・削除ができること（削除は論理削除とする）
- 変更履歴（誰が・いつ・何を変更したか）の保持は必須要件としない。ただし、案件ごとに「最終更新日時」を表示すること

### 2.2 属性データの登録・削除・修正・閲覧

「3. 登録項目」に定める属性データについて、登録・修正・削除・閲覧ができること。

### 2.3 任意条件検索

以下のような条件を組み合わせた検索ができること（AND条件を基本とする）。

- 受付日時（期間指定）
- 対応部署
- 対応状況
- 緊急性
- 地区・場所（住所の一部一致検索、地図上の範囲指定検索を含む）
- キーワード（要望内容の全文検索）

検索結果は一覧表示し、一覧から詳細画面への遷移、CSV出力ができること。

### 2.4 地図表示

- 要望・異常箇所を地図上にピン等で可視化する
- 地図上のピンから詳細情報を参照できる
- 検索結果を地図上にフィルタ表示できる

### 2.5 ユーザー管理・認証

- ID・パスワードによるログイン認証を必須とする（「1.3 想定利用者・権限方針」「4.1 セキュリティ・個人情報保護」参照）
- ユーザーアカウントの発行はシステム管理者権限を持つ職員が行う（職員本人による新規登録は不可）
- ユーザー登録時に、氏名・ユーザーID・所属事務所・担当部署（道路／河川／砂防／その他）・権限区分（一般職員／システム管理者）・初期パスワードを設定する
- パスワード変更ができること。初回ログイン時（初期パスワードのままログインした場合）は変更を必須とし、それ以降は任意のタイミングで変更できること
- システム管理者は、異動・退職等に伴うアカウントの無効化ができること

## 3. 登録項目（属性データ）

個人情報を極力扱わない方針とし、相手方情報は「区分」「要望者」の2項目に絞る。

| 分類 | 項目 | 備考 |
|---|---|---|
| 管理情報 | 受付番号 | 一意なID。自動採番 |
| 管理情報 | 受付日時 | 日付＋時刻を登録する |
| 管理情報 | 受付方法 | 窓口／電話／メール／要望書／FAX／職員パトロール／その他 から選択。「その他」選択時は自由入力欄に具体的な内容を入力する |
| 管理情報 | 登録者（受付職員） | ログインユーザーと連動想定 |
| 管理情報 | 最終更新日時 | 変更履歴は保持しないが、最終更新日時のみ表示する |
| 相手方情報 | 区分 | 個人／自治会／市町村／議員／匿名／職員パトロール／その他 から選択 |
| 相手方情報 | 要望者 | 氏名・団体名等の自由記述。区分が「匿名」「職員パトロール」の場合は入力不要 |
| 案件情報 | 対応部署 | 道路／河川／砂防／その他（土木事務所以外） |
| 案件情報 | 要望の内容 | 自由記述 |
| 案件情報 | 種別 | 苦情／要望／異常発見 の区分 |
| 案件情報 | 要望箇所 | 地図上の座標（緯度経度）＋住所表記 |
| 対応情報 | 対応の必要性 | あり／なし／不明 の3種類 |
| 対応情報 | 緊急性 | 高／中／低 の3種類 |
| 対応情報 | 対応方針 | 自由記述 |
| 対応情報 | 対応状況 | 未対応／対応中／対応完了 |
| 対応情報 | 対応完了日 | 対応状況が「対応完了」になった日 |

- 添付写真・ファイルは今回のスコープ外とする（必要になった時点で別途追加を検討）
- 対応内容・経過記録（対応中の経過を複数回記録する機能）は今回のスコープ外とする

### 3.1 ユーザーアカウント項目

「2.5 ユーザー管理・認証」に関連し、ユーザーアカウントとして以下の項目を管理する。

| 分類 | 項目 | 備考 |
|---|---|---|
| アカウント情報 | ユーザーID | ログインに使用。一意 |
| アカウント情報 | 氏名 | |
| アカウント情報 | パスワード | 初期パスワードはシステム管理者が設定し、初回ログイン時に本人が変更する |
| 所属・権限 | 所属事務所 | ログイン時の事務所判定に使用（「1.3 想定利用者・権限方針」参照） |
| 所属・権限 | 担当部署 | 道路／河川／砂防／その他（土木事務所以外）。編集・削除権限の判定に使用 |
| 所属・権限 | 権限区分 | 一般職員／システム管理者 |
| 所属・権限 | アカウント状態 | 有効／無効化（異動・退職時） |

## 4. 非機能要件

### 4.1 セキュリティ・個人情報保護

- 相手方情報は「区分」「要望者」の2項目のみを扱う方針とし、氏名・住所・連絡先・性別・年齢等の詳細な個人情報は原則保持しない
- クローズドな環境での利用（事務所ごとに管理情報を分離し、他事務所職員は閲覧不可）とするため、ID・パスワードによるログイン認証を必須とする
- 編集・削除権限は対応部署（道路／河川／砂防）に応じて制御する（「1.3 想定利用者・権限方針」参照）
- 通信の暗号化（HTTPS）

### 4.2 可用性・利用環境

- 想定利用場所：インターネット経由でのアクセスを想定
- 想定同時利用者数：最大30人
- 想定データ量：年間最大1,000件程度

### 4.3 保守性・運用

- インフラ方針：本システムはポートフォリオ用途であり、クラウドはAWSを使用する。既存の別アプリ（famigo）用に手動構築済みのRDSインスタンス（`famigo-mysql`、MySQL、db.t4g.micro）に相乗りし、同一インスタンス内に本システム用のデータベース（スキーマ）を新規作成する形でデプロイし、追加コストを極力抑える方針とする
- バックアップ方針：既存RDSインスタンスの自動バックアップ設定（現状：保持期間1日、バックアップウィンドウ19:32-20:02 UTC）にそのまま従う。本システムのために保持期間の変更（延長）は行わない `[要検討: 保持期間1日で要件上十分か。必要なら延長を検討するが、他システム（famigo）と共用のため変更時は影響範囲を確認する]`
- 復旧目標：RPO（復旧時点目標）24時間、RTO（復旧時間目標）1営業日以内
- データの保存期間：登録から3年間保存し、期間経過後は廃棄する
