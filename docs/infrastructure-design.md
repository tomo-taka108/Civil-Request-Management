## 1. 前提・方針

- 本システムはポートフォリオ（転職活動アピール用）作品であり、**追加コストを可能な限り発生させないこと**を最優先とする
- famigoは本システムと無関係の別ポートフォリオアプリ（一般公開中、現在はコスト削減のため停止中・将来再開予定）であり、その本番稼働状況・障害リスクを本システムの設計判断で重く見る必要はない
- 一方で、famigo・本システムのいずれも公開時にブラウザから正常に閲覧・操作できることは必須要件とする
- **EC2・RDSともにfamigoの既存インスタンスに相乗りする**（インスタンスを分けて追加コストを発生させない）。相乗り可否をRDSのみ・EC2のみで分けると、どちらか一方に本システムのインスタンスが見かけ上存在しない状態になり後々の混乱要因になるため、両方相乗りで統一する
- 既存famigo環境は手動構築（ClickOps）されており、TerraformなどのIaC管理下にない。本システムのインフラはTerraformで管理するが、**対象は「本システム用に新規追加する差分」だけでなく、相乗り先であるfamigoの既存リソースを含めた全体像**とする。理由：一部だけをコード化すると、Terraformが認識していない既存リソースとの間で意図しない競合・作成漏れが起きるリスクがあるため（詳細は「4. Terraform化方針」）
- インフラ構築の着手順序：バックエンド・フロントエンドの実装が一定進んだ後に着手する。ただし方針自体は実装着手前の本ドキュメントで先に確定しておき、実装を進める中で判明した点は都度本ドキュメントに追記する

## 3. 相乗り方式の詳細設計

### 3.1 RDS相乗り

`famigo-mysql`インスタンス内に、本システム専用のデータベース（スキーマ）を新規作成する。

```sql
CREATE DATABASE civil_request_management CHARACTER SET utf8mb4 COLLATE utf8mb4_ja_0900_as_cs;
CREATE USER 'civil_request_app'@'%' IDENTIFIED BY '<発行するパスワード>';
GRANT ALL PRIVILEGES ON civil_request_management.* TO 'civil_request_app'@'%';
```

- famigo用のDB・ユーザーとは完全に分離し、本システム用アプリケーションユーザーは自分のスキーマにしかアクセスできないようにする
- インスタンス自体（CPU・メモリ・IOPS・自動バックアップ設定）はfamigoと共用のため、パラメータグループ変更やバックアップ保持期間の変更は行わない（要件定義書4.3参照）
- スキーマ管理はLaravel Migrationで行う（Terraformの管轄外）

### 3.2 EC2相乗り

`famigo-ec2`（t3.micro）上に、famigoアプリと本システムのLaravelアプリを共存させる。

- 想定構成：Nginx（または各アプリのPHP-FPM）で、リクエストのポート or Hostヘッダに応じて振り分け
  - famigo：既存の8080番ポート（変更しない）
  - 本システム：新規ポート（例：8081）でLISTENするPHP-FPM + Nginx server_blockを追加
- t3.microはバースト性能のインスタンスタイプのため、2アプリ同時稼働時のリソース確認は実装・デプロイ段階で要検証（`[要検討]`）
- OS・ミドルウェアのアップデート、再起動を伴う作業はfamigoにも影響するため、実施タイミングに注意する

### 3.3 ALB相乗り方式

現行の443リスナーはルール分岐なし単純forwardのため、以下を新設する。

- 本システム用のターゲットグループを新規作成（例：`civil-request-tg-8081`、ヘルスチェックパスは実装時に確定）
- 443リスナーに新規ルールを追加し、**ホストベースルーティング**で振り分ける（例：`civil-request.<取得予定ドメイン>` → 新ターゲットグループ、それ以外/デフォルト → 既存`famigo-tg-8080`のまま維持）
- 本システム用のドメイン取得・Route53ホストゾーン作成・ACM証明書発行が別途必要（`[要検討: 独自ドメインを新規取得するか、famigo-odekake.comのサブドメインを間借りするかは未確定]`）
- EC2側のSG（`famigo-sg-ec2`）に、新設ポート（例：8081）をALB SGから許可するインバウンドルールを追加

### 3.4 追加コストの見込み

| 項目 | 追加コスト |
|---|---|
| RDSスキーマ追加 | なし（ストレージ逼迫まで無料） |
| EC2アプリ追加 | なし（同一インスタンス内） |
| ALBターゲットグループ・リスナールール追加 | なし（ALB自体は稼働時間課金のみで、ルール数による追加課金なし） |
| Route53ホストゾーン新規作成（独自ドメインを使う場合） | 約0.5USD/月 |
| ドメイン取得（独自ドメインの場合） | ドメイン代（年額、取得先による） |
| ACM証明書 | 無料 |

サブドメイン間借り（`famigo-odekake.com`配下）であればRoute53・ドメイン代も追加0円で完結する。独自ドメインを取得する場合のみ小額のコストが発生する。

## 4. Terraform化方針

### 4.1 スコープ

**famigoの既存リソースを含めた全体像をTerraformでコード化する**（差分のみのコード化は行わない）。

理由：一部のリソース（例：新設するSGルールやターゲットグループ）だけをコード化すると、Terraformはそれらが依存する既存リソース（VPC・サブネット・既存SG等）の実体を認識できない。既存リソースをdata sourceとして参照するだけでは、将来的な変更（例：SG自体の見直し）が発生した際にTerraformで完結できず、手動操作との二重管理になる。ポートフォリオとして「AWS構成をTerraformで一元管理できている」こと自体も見せたい要素であるため、最初から全体をimportする方針とする。

### 4.2 手順

1. 本ドキュメント「2. 既存famigo環境の実地調査結果」を元に、VPC・サブネット・IGW・ルートテーブル・SG（ec2/rds/alb/default）・EC2・RDS・ALB・ターゲットグループ・リスナー・Route53・ACM証明書を`.tf`ファイルとして記述する
2. `terraform import` で1リソースずつ既存の実体をtfstateに取り込む
3. `terraform plan` を実行し、差分ゼロ（コード上の定義と実物が完全一致）になるまでコードを調整する。差分が残ったまま`apply`すると、既存famigoリソースの意図しない再作成・変更が発生するリスクがあるため、**差分ゼロを確認するまでは`apply`しない**
4. 差分ゼロを確認後、本システム用の追加リソース（DBユーザー作成は含まない。SGルール、ターゲットグループ、リスナールール、EC2上のアプリ設定に関連するリソースがあれば等）を`.tf`に追記し、`plan`で追加差分のみであることを確認した上で`apply`する

### 4.3 Terraformの管轄外とするもの

- RDS内のデータベース（スキーマ）・ユーザー作成（`CREATE DATABASE` / `CREATE USER`）：Laravel Migration側で管理
- EC2内のミドルウェア設定（Nginx server_block、PHP-FPM pool設定等）：Ansible等は導入せず、手動 or デプロイスクリプトで管理（`[要検討: 将来的にUser Data / Systems Manager等での自動化を検討]`）

### 4.4 State管理

- tfstateの保管先は`[要検討: S3 + DynamoDBロックか、個人用途のためlocal stateで十分か]`
- 個人ポートフォリオ用途のため、チーム開発を前提としたstate管理の厳格化（リモートバックエンド必須化等）は過剰である可能性がある。まずはlocal stateで進め、必要になった時点でリモート化を検討する
